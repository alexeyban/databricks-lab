## Next steps

### 1. Deploy and validate vault notebooks on Databricks

The vault notebooks in `notebooks/vault/` were generated by the DV 2.0 generator. They need to be
uploaded, run, and verified against live Silver data.

**Tasks:**
- Run `NB_dv_metadata` first — verify hash key helpers execute without errors
- Run `NB_ingest_to_hubs` — verify 13 hub tables created with correct SHA-256 keys
- Run `NB_ingest_to_links` — verify 19 link tables, check hub FK integrity
- Run `NB_ingest_to_satellites` — verify DIFF_HK change detection inserts correctly
- Run `NB_dv_business_vault` — verify PIT daily snapshot and Bridge path tables
- Write a smoke-test script (`skills/docker-databricks-lab-ops/scripts/smoke_test_vault.py`)
  that triggers the vault notebooks and validates row counts

**Acceptance:** Row counts in `vault.hub_*`, `vault.lnk_*`, `vault.sat_*` are non-zero after
running against the full dvdrental Silver dataset.

---

### 2. Wire vault notebooks into the Databricks job

`Orders-ingest-job.yaml` currently ends at Silver. Add the vault stage.

**Tasks:**
- Add tasks for all 5 vault notebooks to `Orders-ingest-job.yaml`:
  - `NB_dv_metadata` runs first (no dependency on Silver tasks)
  - `NB_ingest_to_hubs` depends on `NB_dv_metadata` + all Silver tasks
  - `NB_ingest_to_links` and `NB_ingest_to_satellites` depend on `NB_ingest_to_hubs`
  - `NB_dv_business_vault` depends on both of the above
- Update the job on Databricks via `migrate_and_run.py`
- Verify the full Bronze → Silver → Vault DAG runs end-to-end on schedule

### 4. Run AI classifier with live environment

The `step2b_ai_classifier` is wired in but requires `LLM_API_KEY` + `DATABRICKS_WAREHOUSE_ID`.

**Tasks:**
- Set env vars and run a fresh `--analyze` without `--no-ai`
- Inspect `02b_merged_classification.json` — verify AI and heuristics agree on dvdrental
- Check `decisions.log` for any `MERGE:heuristic_only` or `MERGE:ai_only` discrepancies
- Tune the AI system prompt if disagreements are semantically incorrect

**Expected outcome:** For dvdrental, zero LOW-confidence entities (AI and heuristics should fully agree on a well-known schema).

### 5. dbt Gold layer expansion

Currently Gold has only `gold_film` and `gold_rental`. The vault layer enables richer business models.

**Tasks:**
- Add `gold_customer` — customer lifetime value, rental history, payment totals
- Add `gold_inventory` — stock levels, utilisation rate per store
- Add `gold_staff_performance` — rental/payment counts per staff member
- Add `gold_film_popularity` — rental frequency, revenue per film
- Extend data quality tests (`dbt test`) to cover new models
- Document lineage: Silver → Vault → Gold for each new model

---

### 6. Vault monitoring

No observability exists for the vault layer yet.

**Tasks:**
- Add `workspace.monitoring.vault_load_log` Delta table: one row per vault notebook run
  (hub/link/sat name, rows_inserted, load_date, duration_s, status)
- Extend `NB_schema_drift_helpers` to cover vault layer schema changes
- Add a Grafana / Databricks SQL dashboard showing hub/link/sat row counts over time

### 7. Generator — re-usability for other source schemas

The DV 2.0 generator is currently wired to `pipeline_configs/silver/dvdrental/`. Making it
schema-agnostic unlocks its use for any new CDC source.

**Tasks:**
- Parameterise `--config-dir` fully (already done) — verify it works for a different Silver schema
- Add a `generators/dv_generator/export_silver_config.py` helper that introspects a live
  Databricks Silver schema and writes the `pipeline_configs/silver/<source>/*.json` input files
  automatically (removing the need to hand-write them)
- Document the end-to-end flow for a net-new source in `design/dv2/DV2_GENERATOR_DESIGN.md`
